Chapter 00: Project Kickstart & Vision

0.1: The Final Product

Welcome! In this project, we will build a beautiful, functional Habit Tracker app from scratch. Here's a preview of what you'll create.

0.3: Project Creation & Folder Setup

1. Run this command in your terminal:

flutter create habit_tracker

2. Open the `habit_tracker` folder. Inside `lib`, create a `controllers` folder. Our structure is now:

lib/
├── controllers/    # For our GetX controllers
├── models/         # For our data structures
├── screens/        # For each full screen of our app
├── widgets/        # For small, reusable UI components
└── main.dart

Why do we create this folder structure?

Organizing code makes it easier to find things and work efficiently. This structure separates our code based on its purpose, which is a core principle of good software design. It makes the project much easier to understand and scale up later.

Chapter 01: The Main Screen - A Static View

In this chapter, we will build the visual shell of the main screen. It won't have any logic yet, but it will look correct. This "static-first" approach is a great way to build UIs.

1.1: Add the GetX Package

Open your `pubspec.yaml` file and add `get` under dependencies. We do this now because our `main.dart` file will use a widget from this package.

pubspec.yaml
dependencies:
  flutter:
    sdk: flutter
  get: ^4.6.6 # Add this line

After adding the line, run `flutter pub get` in your terminal to download and install the package.

1.2: The Habit Model

First, define what a "habit" is in our code. This is our data model.

lib/models/habit.dart
class Habit {
  final String name;
  final String description;
  bool isCompleted;

  Habit({
    required this.name,
    required this.description,
    this.isCompleted = false,
  });
}

Why create a Model class?

A Model defines the "shape" of our data. By creating a `Habit` class, we ensure that every habit in our app has a consistent structure. This prevents typos and makes our code self-documenting.

1.3: The HabitListItem Widget

Next, we'll create a dedicated, reusable widget to display a single habit. It will need a function to call when the checkbox is tapped.

lib/widgets/habit_list_item.dart
import 'package:flutter/material.dart';

class HabitListItem extends StatelessWidget {
  final String habitName;
  final bool isCompleted;
  final Function(bool?)? onToggle;

  const HabitListItem({
    super.key,
    required this.habitName,
    required this.isCompleted,
    required this.onToggle,
  });

  @override
  Widget build(BuildContext context) {
    return ListTile(
      title: Text(habitName),
      leading: Checkbox(
        value: isCompleted,
        onChanged: onToggle,
      ),
    );
  }
}

Why build a separate Widget?

This makes our code reusable (we can use this item anywhere), easier to read, and easier to debug. If there's a problem with how a habit looks, we know exactly which file to check!

1.5: Update main.dart

Finally, tell our app to show our `MainScreen` on launch. We'll use `GetMaterialApp` to prepare for using GetX in the next chapter.

}

Run the app now. You should see your static main screen. It doesn't do anything yet, but we have built the visual foundation. In the next chapter, we'll make it dynamic.

Chapter 02: Reactive State with GetX

This is where the magic happens! We'll introduce GetX to manage our app's state. This means our UI will automatically update when our data changes, without needing `StatefulWidget` or `setState()`.

2.1: Create the HabitController

The Controller separates our logic from our UI. Notice the .obs - this makes our list "reactive."

lib/controllers/habit_controller.dart
import 'package:get/get.dart';
import '../models/habit.dart';

class HabitController extends GetxController {
  var habits = [].obs;

  @override
  void onInit() {
    super.onInit();
    habits.addAll([
      Habit(name: 'Read for 15 minutes', description: 'Expand the mind'),
      Habit(name: 'Go for a walk', description: 'Good for the body', isCompleted: true),
      Habit(name: 'Drink 8 glasses of water', description: 'Stay hydrated'),
    ]);
  }

  void toggleHabit(Habit habit) {
    habit.isCompleted = !habit.isCompleted;
    habits.refresh();
  }
}

Why use a Controller?

The Controller separates our business logic (what the app *does*) from our UI code (what it *looks like*). Our UI widgets become simple and "dumb"—their only job is to display data. This separation is crucial for building large, testable, and maintainable applications.

2.2: Refactor the Main Screen

We'll now connect our `MainScreen` to the `HabitController`. Get.put() creates our controller. The Obx widget rebuilds its child whenever the `habits` list changes.

lib/screens/main_screen.dart
import 'package:flutter/material.dart';
import 'package:get/get.dart';
import '../controllers/habit_controller.dart';
import '../widgets/habit_list_item.dart';

class MainScreen extends StatelessWidget {
  const MainScreen({super.key});

  @override
  Widget build(BuildContext context) {
    final HabitController controller = Get.put(HabitController());

    return Scaffold(
      appBar: AppBar(title: const Text('My Habits (GetX)')),
      body: Obx(() => ListView.builder(
        itemCount: controller.habits.length,
        itemBuilder: (context, index) {
          final habit = controller.habits[index];
          return HabitListItem(
            habitName: habit.name,
            isCompleted: habit.isCompleted,
            onToggle: (value) => controller.toggleHabit(habit),
          );
        },
      )),
      floatingActionButton: FloatingActionButton(
        onPressed: () {},
        child: const Icon(Icons.add),
      ),
    );
  }
}

Milestone Reached!

Run the app. The UI now loads its data from the controller and updates instantly when you tap a checkbox. This is the power of reactive programming!

GetX is a powerful and lightweight framework for Flutter. It's more than just a state manager; it handles dependency injection, route management, and more.

Its primary goal is to increase developer productivity by reducing boilerplate code without sacrificing performance.

Why we are using it: Instead of manually managing state with `StatefulWidget` and `setState()`, GetX allows us to create reactive variables. When these variables change, any widget listening to them automatically rebuilds. This makes our code cleaner, more organized, and easier to maintain.

The .obs is a GetX extension that turns any variable into a "reactive" or "observable" stream.

When you have a variable like `var name = 'John'.obs;`, you are creating a special stream that holds the value 'John'. GetX's reactive widgets (like `Obx`) can subscribe to this stream.

Any time you change the value of `name` (e.g., `name.value = 'Doe'`), the stream fires an event, and all listening widgets automatically update themselves with the new value. For lists, you call `.refresh()` to tell the stream that the contents of the list have changed.

Get.put() is GetX's way of performing dependency injection. When you call `Get.put(HabitController())`, you are telling GetX:

  • "Create an instance of `HabitController` for me."
  • "Keep this instance in memory so I can access it from anywhere in my app."
  • "Manage its lifecycle. When this screen is no longer used, you can intelligently remove it from memory to prevent leaks."

This is incredibly powerful because it decouples our UI from the controller's creation. We don't have to pass the controller instance down through multiple widgets. Any child widget can simply call `Get.find()` to get access to the same instance.

Obx is one of GetX's primary reactive widgets. It takes a single function as its child, and that function must return a widget.

The magic of `Obx` is that it automatically subscribes to any .obs variables that are used inside its function. In our case, it sees that we are using `controller.habits.length`, so it starts "listening" to the `habits` list.

Whenever `habits` is changed (an item is added, removed, or we call `.refresh()`), `Obx` will automatically re-run its builder function, thus rebuilding the `ListView` with the new data. It is highly efficient because it only rebuilds this specific part of the UI, not the entire screen.

Chapter 02: Reactive State with GetX

We'll introduce GetX to manage our app's state, replacing `StatefulWidget`.

2.1: Create the HabitController

The GetxController separates logic from UI. The .obs makes our list "reactive."

lib/controllers/habit_controller.dart
import 'package:get/get.dart';
import '../models/habit.dart';

class HabitController extends GetxController {
  var habits = [].obs;

  @override
  void onInit() {
    super.onInit();
    habits.addAll([
      Habit(name: 'Read for 15 minutes', description: 'Expand the mind'),
      Habit(name: 'Go for a walk', description: 'Good for the body', isCompleted: true),
    ]);
  }

  void toggleHabit(Habit habit) {
    habit.isCompleted = !habit.isCompleted;
    habits.refresh();
  }
}

2.2: Refactor the Main Screen

We'll now connect our `MainScreen` to the `HabitController`. Get.put() creates our controller. The Obx widget rebuilds its child whenever the `habits` list changes.

lib/screens/main_screen.dart
import 'package:flutter/material.dart';
import 'package:get/get.dart';
import '../controllers/habit_controller.dart';
import '../widgets/habit_list_item.dart';

class MainScreen extends StatelessWidget {
  const MainScreen({super.key});

  @override
  Widget build(BuildContext context) {
    final HabitController controller = Get.put(HabitController());

    return Scaffold(
      appBar: AppBar(title: const Text('My Habits (GetX)')),
      body: Obx(() => ListView.builder(
        itemCount: controller.habits.length,
        itemBuilder: (context, index) {
          final habit = controller.habits[index];
          return HabitListItem(
            habitName: habit.name,
            isCompleted: habit.isCompleted,
            onToggle: (value) => controller.toggleHabit(habit),
          );
        },
      )),
      floatingActionButton: FloatingActionButton(
        onPressed: () {},
        child: const Icon(Icons.add),
      ),
    );
  }
}

Milestone Reached!

Run the app. The UI now loads its data from the controller and updates instantly when you tap a checkbox! This is the power of reactive programming!

class AddEditHabitScreen extends StatelessWidget {

  Widget build(BuildContext context) {

    final descriptionController = TextEditingController();
    // Find the existing controller instance created on the MainScreen
    final HabitController controller = Get.find();

    return Scaffold(
      appBar: AppBar(title: const Text('Add New Habit')),
      body: Padding(
        padding: const EdgeInsets.all(16.0),
        child: Column(
          children: [
            TextField(
              controller: nameController,
              decoration: const InputDecoration(labelText: 'Habit Name'),
            ),
            const SizedBox(height: 16),
            TextField(
              controller: descriptionController,
              decoration: const InputDecoration(labelText: 'Description'),

                // Call the controller method

                  nameController.text,
                  descriptionController.text,
                );
                // Go back to the previous screen
                Get.back();
              },
              child: const Text('Save Habit'),
            ),
          ],
        ),
      ),
    );
  }

Chapter 04: Data Persistence

Our app works well, but there's a huge problem: if you close the app, all your habits disappear! This is because they only exist in the phone's memory. In this chapter, we'll fix this by saving the habits to the device's storage, so the data persists between sessions.

4.1: Add & Initialize GetStorage

We will use GetStorage, a fast key-value storage library that's part of the GetX ecosystem.

1. First, add it to your dependencies:

pubspec.yaml
dependencies:
  flutter:
    sdk: flutter
  get: ^4.6.6
  get_storage: ^2.1.1 # Add this line

2. Next, initialize `GetStorage` in your `main` function before you run the app. This needs to be an `async` function now.

lib/main.dart
import 'package:flutter/material.dart';
import 'package:get/get.dart';
import 'package:get_storage/get_storage.dart'; // Import this
import 'screens/main_screen.dart';

// Make main async and initialize storage
Future main() async {
  await GetStorage.init();
  runApp(const MyApp());
}

class MyApp extends StatelessWidget {
  const MyApp({super.key});

  @override
  Widget build(BuildContext context) {
    return GetMaterialApp(
      title: 'Habit Tracker',
      theme: ThemeData.dark(),
      home: const MainScreen(),
    );
  }
}

4.2: Prepare the Model for Storage

GetStorage saves data in a key-value format. To make our `Habit` class compatible, we must teach it how to serialize itself into a storable Map and how to deserialize itself from a map back into a `Habit` object. We do this by creating a `toJson()` method and a special factory constructor called `fromJson()`.

lib/models/habit.dart
class Habit {
  final String name;
  final String description;
  bool isCompleted;

  Habit({
    required this.name,
    required this.description,
    this.isCompleted = false,
  });

  // Method to convert a Habit instance to a Map (storable format)
  Map toJson() => {
    'name': name,
    'description': description,
    'isCompleted': isCompleted,
  };

  // Factory constructor to create a Habit instance from a Map
  factory Habit.fromJson(Map json) => Habit(
    name: json['name'],
    description: json['description'],
    isCompleted: json['isCompleted'],
  );
}

4.3: Update the HabitController

This is the final step. We'll update the controller to use `GetStorage`. It will now load habits from storage when it starts and automatically save the habit list whenever it changes.

lib/controllers/habit_controller.dart
import 'package:get/get.dart';
import 'package:get_storage/get_storage.dart';
import '../models/habit.dart';

class HabitController extends GetxController {
  var habits = [].obs;
  final box = GetStorage();

  @override
  void onInit() {
    super.onInit();
    
    // Read the stored habits from the 'habits' key.
    List? storedHabits = box.read('habits');
    
    // If there are stored habits, parse them from JSON and load them.
    if (storedHabits != null) {
      habits.value = storedHabits.map((e) => Habit.fromJson(e)).toList();
    }
    
    // The dummy data from Chapter 2 is no longer needed, as we now
    // persist the user's actual data.

    // Use a GetX worker called 'ever'. It listens to the 'habits' list.
    // Every time the list is changed (add, remove, update), it saves
    // the entire list to storage.
    ever(habits, (_) {
      box.write('habits', habits.toList().map((e) => e.toJson()).toList());
    });
  }

  void addHabit(String name, String description) {
    if (name.isEmpty) return;
    final newHabit = Habit(name: name, description: description);
    // When we add a habit here, the 'ever' worker will automatically save it.
    habits.add(newHabit);
  }

  void toggleHabit(Habit habit) {
    habit.isCompleted = !habit.isCompleted;
    // When we toggle here, the 'ever' worker will also save the change.
    habits.refresh();
  }
}

Why use an `ever` worker?

The ever worker from GetX is incredibly powerful. It triggers a callback function *every time* the reactive variable (`habits`) changes. By using it, we no longer need to manually call a "save" function inside `addHabit` and `toggleHabit`. The saving happens automatically in one central place, making our code cleaner and less error-prone.

Persistence Achieved!

Run the app. Add a new habit or toggle an existing one. Now, completely close the app from your device's recent apps list and reopen it. Your changes will still be there! Your app now has a memory.

The pubspec.yaml file is the manifest for your Flutter project. It's the central configuration file that defines the project's metadata and dependencies.

Key Sections

  • dev_dependencies: Lists packages only needed for development and testing (e.g., flutter_lints).
  • flutter: Contains Flutter-specific configurations, like declaring assets (images, fonts).

The lib (short for library) folder is the heart of your Flutter project. It's where all your Dart code for the application lives.

Key Files & Folders

  • main.dart: The entry point of your application. Execution starts here.
  • Sub-folders (e.g., screens, models): Used to organize your code by feature or type, making the project maintainable.

A class is a blueprint for creating objects. It bundles data (properties) and functions (methods) that operate on that data into a single, reusable unit.

Example

// Defines the blueprint for a 'Habit'
class Habit {
  // Properties (data)
  final String name;
  bool isCompleted;

  // Constructor (how to create a Habit object)
  Habit({required this.name, this.isCompleted = false});

  // Method (function)
  void toggle() {
    isCompleted = !isCompleted;
  }
}

The final keyword declares a variable that can only be assigned a value once. It must be initialized either when it's declared or in the class constructor.

Why Use 'final'?

  • Immutability: It helps create objects whose state cannot be changed after creation, leading to more predictable and less error-prone code.
  • Safety: Prevents accidental reassignment of a variable that is intended to hold a constant value throughout the object's life.

A StatelessWidget is a widget that describes a part of the user interface which cannot change its own state once built. Its appearance is determined solely by the configuration passed into it from its parent.

When to Use

  • For static content like an icon, a label, or a screen that just displays information.
  • When the widget only needs to be redrawn if its parent provides new data.

A ListTile is a convenient, pre-styled Material Design widget that creates a fixed-height row. It's perfectly designed for use in lists.

Common Properties

  • leading: A widget to display before the title (e.g., an Icon or Checkbox).
  • title: The primary content of the list tile, typically a Text widget.
  • subtitle: Additional content displayed below the title.
  • trailing: A widget to display after the title (e.g., an arrow icon).

Key Properties

Key Properties

  • appBar: The horizontal bar displayed at the top of the screen.
  • body: The primary content of the scaffold, displayed below the appBar.
  • floatingActionButton: A button that "floats" over the body, often used for a primary action.

An AppBar is the toolbar that appears at the top of a screen, typically within a Scaffold. It's used for branding, screen titles, and actions.

Key Properties

  • title: The main widget displayed in the app bar, usually a Text widget.
  • leading: A widget to display at the beginning of the app bar (often a back button or menu icon).
  • actions: A list of widgets to display after the title (e.g., settings or search icons).

A ListView is a core widget that arranges its children sequentially in a scrollable list. The standard constructor is best for a small, fixed number of children because it builds all of them at once.

For long lists, use ListView.builder

For lists with many items, ListView.builder is much more performant because it only builds the items that are currently visible on screen.

GetMaterialApp is GetX's replacement for Flutter's standard MaterialApp. It acts as the root of your application, enabling all of GetX's powerful features for routing, state, and dependency management.

ThemeData is a class that holds the color and typography values for a theme. Defining a ThemeData at the top of your app (in GetMaterialApp) ensures all child widgets inherit the same visual styling for a consistent look and feel.

A GetxController is a class from GetX designed to separate your business logic and state variables from your UI code, promoting a clean architecture.

Core Features

  • State Management: Holds reactive variables (e.g., .obs) that automatically update the UI.
  • Lifecycle Methods: Provides methods like onInit() (setup) and onClose() (cleanup).
  • Dependency Injection: Easily created and accessed using Get.put() and Get.find().

Get.put() is GetX's primary dependency injection method. It creates an instance of a controller (like HabitController) and registers it with GetX, making it available to all child widgets in the widget tree.

The .obs is a GetX extension that turns any variable into a "reactive" or "observable" stream. When an .obs variable's value changes, any reactive widget (like Obx) that uses it will automatically rebuild.

Example

// A normal integer
var count = 0;

// A reactive integer
var reactiveCount = 0.obs;

// To change the value
reactiveCount.value = 1;
// or simply
reactiveCount++;

Obx is a GetX reactive widget that takes a function returning a widget. It automatically subscribes to any .obs variables used within that function. Whenever one of those variables changes, Obx efficiently rebuilds only itself, not the whole screen.

Get.to() is GetX's primary method for navigation. It pushes a new screen onto the navigation stack. It's simpler than Flutter's native Navigator.push() because it doesn't require the BuildContext.

A TextField is a fundamental widget that allows users to enter text via a keyboard. It's the primary building block for creating forms in Flutter.

Key Properties

  • controller: A TextEditingController to manage the text.
  • decoration: An InputDecoration to style the text field with labels, hints, and borders.

A TextEditingController is an object used to manage the text of a TextField. You attach a controller to a TextField to be able to read the text the user has entered (`controller.text`), set the text programmatically, or listen for changes.

The Padding widget is a simple but essential layout widget. Its only job is to add space around its child widget, defined by an EdgeInsets value.

The Column widget is a layout widget that arranges its children in a vertical array. It's one of the most common widgets for building UIs.

An ElevatedButton is a Material Design button that appears "elevated" with a shadow. It's typically used for high-priority actions in your app, like "Save" or "Submit."

The onPressed property is a required callback function for most button widgets. It defines the code that will be executed when the user taps the button. If you pass null to onPressed, the button will be automatically disabled.

Get.back() is the GetX method for closing the current screen and "popping" it off the navigation stack, returning the user to the previous screen. It's the counterpart to Get.to().

Get.find() is the GetX method used to find an already-created instance of a controller that was previously registered with Get.put(). It allows any widget, anywhere in the widget tree, to get access to the same controller instance without needing it to be passed down manually.

A FloatingActionButton (FAB) is a circular icon button that "floats" above the UI, typically in the bottom-right corner. It's used for a primary, promoted action on a screen, such as adding a new item.

GetStorage is a fast, extra-light, and synchronous key-value storage library. It's written entirely in Dart and integrates perfectly with the GetX ecosystem.

Why Use GetStorage?

  • Simplicity: The API is extremely simple: `box.write(key, value)` and `box.read(key)`.
  • Performance: It's significantly faster than other key-value solutions like `shared_preferences`.
  • No `async` needed: After initialization, all read/write operations are synchronous, which simplifies your code.

Serialization is the process of converting a complex data object (like our `Habit` class instance) into a format that can be easily stored or transmitted, and then reconstructed later. The most common format for this is JSON.

The toJson() method is a convention used in Dart for serialization. Its job is to convert an instance of a class into a `Map`, which is a structure that can be easily encoded into a JSON string.

A fromJson() factory constructor is the reverse of `toJson()`. It takes a `Map` (decoded from JSON) and uses it to construct a new instance of the class, populating its properties with the map's values.

The ever worker is a feature of GetX that listens to a reactive variable (an `.obs` variable) and executes a callback function every time that variable's value changes.

Why is it useful here?

By using `ever(habits, ...)`, we create a single, reliable listener that automatically saves our entire list to storage whenever *any* change occurs (adding, removing, or updating a habit). This is much cleaner than adding a `box.write()` call in every single method that modifies the list.

A Map is a collection of key-value pairs. Each key in a map must be unique. It's Dart's equivalent of a dictionary or hash table in other languages, and it's the perfect structure for representing JSON data.

Syntax

  • Declaration: A map is declared using curly braces {}.
  • Example: var myMap = {'name': 'Alice', 'age': 30};
  • Type Annotation: We often use Map for JSON, meaning the keys are always strings, but the values can be of any type (String, int, bool, etc.).

A factory constructor is a special type of constructor that gives you more control over the object creation process. Unlike a normal constructor, a factory constructor does not always create a new instance of its class.

Key Use Cases

  • Returning a cached instance: It can return an existing instance from a cache instead of creating a new one.
  • Returning a subtype instance: It can decide to return an instance of a subclass.
  • Complex initialization logic: It's perfect for creating an instance from a different data structure, like a Map. This is why we use it for `fromJson`, as it allows us to parse the map and return a fully-formed `Habit` object.

`super.key` is a modern Dart feature that forwards a widget's `Key` to its parent class constructor (`super`). A Key is an object used by Flutter's engine to identify and differentiate widgets.

Why are Keys important?

  • State Preservation: When widgets in a list are reordered, deleted, or added, Flutter uses keys to understand which widget is which, ensuring the correct state is preserved.
  • Performance: Using keys helps Flutter's diffing algorithm to be more efficient when updating the UI. While not always necessary for simple static lists, it's a critical best practice, especially for lists that can change.

A SizedBox is a simple but highly efficient widget whose only purpose is to create a box with a specified size. It is most commonly used to create fixed-size gaps or spacing between other widgets.

Why use SizedBox?

  • Performance: It's more performant than using other layout widgets like `Padding` or `Container` just to create space, as it's a very simple `const` widget with no complex painting or layout calculations.
  • Clarity: Using `const SizedBox(height: 16)` is very clear and explicitly communicates the intent of creating vertical space.

`async` and `await` are keywords that provide a declarative way to handle asynchronous operations in Dart. They help you write code that looks synchronous but works asynchronously, avoiding complex callback chains (often called "callback hell").

How they work

  • async: You mark a function body with the `async` keyword to indicate that it is asynchronous. This allows you to use `await` inside it.
  • Future: Asynchronous functions immediately return a `Future`, which is a promise of a value to be delivered later.
  • await: You use `await` in front of a call to an async function. It pauses execution of the current function *until the `Future` completes* and returns its result. In our case, `await GetStorage.init()` pauses the `main` function until the storage is fully initialized before trying to run the app.

`onInit()` is a lifecycle method provided by `GetxController`. It is called automatically by GetX right after the controller is allocated in memory and registered by `Get.put()`.

When to use it

  • Initialization Tasks: It's the perfect place to perform setup logic for your controller.
  • Loading Data: Fetching initial data from a database, local storage (`GetStorage`), or an API.
  • Setting up Listeners: Initializing workers like `ever()` or other subscriptions that need to exist for the lifetime of the controller.

The `refresh()` method is a way to manually force a reactive variable in GetX to update and notify its listeners (like the `Obx` widget).

Why is it needed here?

  • Complex Object Changes: When you have a list of objects (like `var habits = [].obs`), GetX's reactive system can detect when you add or remove items from the list (`habits.add(...)`).
  • Internal Property Changes: However, it cannot automatically detect changes to a property *inside* one of the objects in the list (e.g., `habit.isCompleted = true`). The list itself hasn't changed, only a property of one of its items.
  • Forcing an Update: Calling `habits.refresh()` tells GetX: "Hey, something deep inside the `habits` list has changed. Please notify all widgets listening to this list so they can rebuild with the new data."
